## Why this matters

`backend/pyproject.toml` lists ~100 packages in a single flat
`[project.dependencies]` array — effectively a `pip freeze` dump. It mixes true
runtime deps (FastAPI, motor, openai, boto3, pillow, reportlab) with test and
load-test tooling (pytest, pytest-*, faker, **locust**, **flask**, **gevent**,
**werkzeug**, flask-cors, flask-login, locust-cloud, pyzmq, python-socketio …).
Production deployments therefore ship an entire Flask/locust load-testing stack
they never run, enlarging the image and the attack surface. Splitting test/dev
tooling into `[project.optional-dependencies]` makes production installs lean
and the dependency intent legible.

## Steps

### Step 1: Define the optional dependency groups

Edit `backend/pyproject.toml`. Remove the top-level test/load-test packages
listed in Current state from `[project.dependencies]`, and add a new table.
Keep version pins where you move them. Target shape:

```toml
[project.optional-dependencies]
test = [
    "pytest==8.4.1",
    "pytest-asyncio==1.1.0",
    "pytest-cov==6.2.1",
    "pytest-mock==3.14.1",
    "faker==37.4.2",
    "httpx==0.28.1",          # used by tests; if imported by runtime code, keep in base instead
]
load = [
    "locust==2.37.14",
    "locust-cloud==1.26.3",
]
dev = [
    "auto-author-backend[test,load]",
]
```

Do NOT individually list transitive deps of pytest/locust (werkzeug, gevent,
greenlet, flask, pyzmq, zope-*, etc.) — delete those lines from
`[project.dependencies]` and let the resolver pull them in via `locust`/`pytest`
when the extras are installed. Before deleting a line, confirm it is a
transitive of the test/load stack and not a runtime dep (when unsure, check
whether any `backend/app/**` file imports it: `grep -rn "import <pkg>" backend/app`).

**Decision — httpx**: if `grep -rn "import httpx\|from httpx" backend/app`
returns matches, httpx is a runtime dep → keep it in `[project.dependencies]`
and remove it from the `test` extra. Otherwise leave it in `test`.

### Step 2: Re-resolve and lock with extras

`cd backend && uv sync --extra test --extra load` (or `uv sync --all-extras`)
to regenerate `uv.lock` with the new structure.

**Verify**: `cd backend && uv run pytest tests/ --collect-only -q` → collects
(pytest is available via the test extra).

### Step 3: Confirm a runtime-only install excludes the test/load stack

Produce a runtime-only resolution and confirm Flask/locust are absent. Using
uv's export:
`cd backend && uv export --no-dev --no-emit-project > /tmp/runtime-reqs.txt`
(or `uv export` without extras — use whichever your uv version supports).

**Verify**:
`grep -iE "^(flask|locust|gevent|werkzeug|pytest)" /tmp/runtime-reqs.txt` →
no matches (these appear only when extras are included).

### Step 4: Confirm the app still imports/boots

**Verify**: `cd backend && uv run python -c "import app.main"` → exits 0 (the
runtime imports resolve without the test/load packages, because nothing in
`app/` imports them).
